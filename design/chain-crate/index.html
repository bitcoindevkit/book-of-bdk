Chain Crate: The Foundation for Wallet State¶

Introduction¶

The bdk_chain crate is the foundational data structure library at the heart of Bitcoin Dev Kit. While BDK's wallet libraries provide high-level interfaces for sending and receiving bitcoin, the chain crate solves a more fundamental problem: how do you maintain consistent, reliable state about what's happening on the bitcoin blockchain?

This is harder than it might seem at first glance. A Bitcoin wallet needs to:

  • Track which blocks exist in its view of the blockchain
  • Handle chain reorganizations gracefully
  • Store transactions and understand their relationships
  • Manage conflicts (multiple transactions spending the same input)
  • Know which outputs belong to the wallet
  • Determine confirmation status reliably
  • Support multiple data sources (electrum, esplora, full nodes, etc.)
  • Work with different persistence backends
  • Never end up in an inconsistent state

The chain crate provides the primitives to solve all of these problems while remaining data source agnostic and persistence agnostic. It doesn't care where you get blockchain data from or how you store it—it simply provides robust mechanisms for processing that data consistently.

Design Philosophy¶

Monotonicity: The Foundation of Reliability¶

The cornerstone of the chain crate's design is monotonicity—the principle that data structures only grow and never shrink. This might seem limiting at first, but it provides enormous benefits:

  1. No inconsistent states: You can't accidentally delete a transaction and leave references to it elsewhere
  2. Trivial merging: Two instances of the same structure can always be merged without conflicts
  3. Predictable recovery: After a crash, you can replay changes without worrying about order
  4. Time-travel debugging: You can reconstruct state at any point in history

In practice, this means:

  • You can always add transactions, but never delete them
  • You can add new checkpoints to the chain, but the genesis block is immutable
  • You can add new script pubkeys to index, but never remove them
  • You can add new anchors (confirmations), but they accumulate over time

This monotone property makes the entire system remarkably robust. Even if your application crashes mid-update, the state remains consistent.

The ChangeSet Pattern: Event Sourcing for Bitcoin Wallets¶

Every state-modifying operation in the chain crate returns a ChangeSet object. A ChangeSet represents the delta between two states—exactly what changed.

let changeset = tx_graph.insert_tx(new_transaction);

These ChangeSets can be:

  • Persisted to storage (enabling crash recovery)
  • Applied to another instance (syncing state between systems)
  • Merged together (combining multiple updates)
  • Used for event notifications (triggering UI updates)

This is essentially an event sourcing pattern. Your entire wallet state can be reconstructed by replaying all ChangeSets from the beginning. This makes the system:

  • Debuggable: You can inspect exactly what changed and when
  • Auditable: You have a complete history of state transitions
  • Recoverable: Lost state can be rebuilt from stored ChangeSets
  • Testable: You can verify that specific inputs produce expected ChangeSets

The pattern also enables the crate to remain persistence agnostic. The crate produces ChangeSets; your application decides how to store them (SQLite, files, memory, etc.).

Separation of Concerns: Three Distinct Responsibilities¶

The chain crate recognizes that a bitcoin wallet has three orthogonal concerns:

  1. Chain tracking (LocalChain): Which blocks exist and in what order?
  2. Transaction tracking (TxGraph): Which transactions exist and how do they relate?
  3. Output indexing (Indexer): Which outputs are "mine"?

These are kept deliberately separate because they have different lifecycles and requirements:

  • The chain can reorganize independently of transactions
  • Transactions can arrive before you know which block they're in
  • The indexer might discover new addresses that require re-scanning historical transactions

By keeping these concerns separate, the crate provides flexibility. You could:

  • Use TxGraph without an indexer (for non-wallet applications)
  • Use SpkTxOutIndex without TxGraph (for address monitoring)
  • Use LocalChain standalone (for block explorers)
  • Combine them in custom ways for specialized use cases

Core Abstractions¶

LocalChain: Tracking Blockchain State¶

The LocalChain structure maintains your local view of the Bitcoin blockchain using a chain of checkpoints:

pub struct LocalChain {
    tip: CheckPoint,
}

Each checkpoint is simply a block height and hash pair. The chain structure is elegant: - The genesis block is required and immutable - Checkpoints form a linked list from tip back to genesis - You don't need every block—sparse checkpoints work fine

What it provides:

The key service LocalChain provides is implementing the ChainOracle trait, which answers the question: "Is block X in the chain leading to tip Y?" This seemingly simple query is crucial for determining transaction confirmation status.

Handling reorganizations:

When you apply an update that conflicts with the current chain (a reorg), LocalChain automatically: 1. Finds the common ancestor 2. Keeps all checkpoints up to that point 3. Applies the new chain from there

The update method returns the ChangeSet representing this operation, which includes information about what was replaced. Your application can use this to update transaction anchors accordingly.

Design rationale:

Why checkpoints instead of full blocks? Because wallets don't need full block data—they only need enough information to: - Verify transaction confirmations - Detect reorganizations - Understand chain structure

Storing just height/hash pairs keeps memory usage minimal while providing all necessary functionality.

TxGraph: The Transaction Graph at the Heart of BDK¶

The TxGraph is the most sophisticated component in the chain crate. It maintains a graph of transactions and their relationships:

pub struct TxGraph<A> {
    txs: HashMap<Txid, TxNodeInternal>,
    spends: BTreeMap<OutPoint, HashSet<Txid>>,
    anchors: HashMap<Txid, BTreeSet<A>>,
    first_seen: HashMap<Txid, u64>,
    last_seen: HashMap<Txid, u64>,
    last_evicted: HashMap<Txid, u64>,
    // ... optimized indexes
}

Key insight: Allowing conflicts

The most important design decision in TxGraph is that it allows conflicting transactions to coexist. Two transactions spending the same input can both exist in the graph simultaneously.

This might seem counterintuitive—doesn't Bitcoin's consensus rules mean only one can be valid? Yes, but at any given moment, you might observe: - Multiple unconfirmed transactions competing in the mempool - A confirmed transaction and a conflicting mempool transaction (during a reorg) - Historical conflicts that help you understand what happened

Rather than forcing applications to choose winners immediately, TxGraph stores all observed transactions and defers conflict resolution to canonicalization time, when you have full context to make the right choice.

Full transactions and partial transactions

The graph can store: - Full transactions: Complete transaction data - Partial transactions: Just the outputs (TxOut)

Partial transactions are useful because: - You need input values for fee calculation, but don't need full input transactions - During scanning, you might only extract relevant outputs for efficiency - You can fill in full details later as you learn more

The graph will merge data: if you insert a partial transaction and later insert the full version, it updates seamlessly.

Generic anchors: Flexible confirmation tracking

The <A> type parameter represents an "anchor"—proof that a transaction exists at some blockchain position. The crate provides: - BlockId: Simple height + hash - ConfirmationBlockTime: Adds confirmation timestamp

But you can define custom anchor types that include: - Merkle proofs - Multiple confirmation sources - Confidence scores - Whatever metadata your application needs

The only requirement is implementing the Anchor trait:

pub trait Anchor {
    fn anchor_block(&self) -> BlockId;
    fn confirmation_height_upper_bound(&self) -> u32;
}

This generic design means TxGraph works for everything from simple wallets to applications that need sophisticated confirmation tracking.

Tracking mempool state

The graph maintains three timestamp maps: - first_seen: When you first observed this transaction - last_seen: Most recent observation (updated on every insert) - last_evicted: When you saw it evicted from mempool

These timestamps play a crucial role in conflict resolution when transactions lack anchors.

Graph traversal

TxGraph provides methods for walking the transaction graph: - walk_ancestors(): Find all transactions this one depends on - walk_descendants(): Find all transactions that depend on this one - Finding conflicts (transactions spending the same inputs) - Calculating transaction chains

This enables applications to: - Calculate fees accurately (need input values) - Detect unconfirmed chains - Understand transaction relationships - Handle CPFP (child pays for parent)

Canonicalization: Resolving Conflicts¶

Since TxGraph allows conflicts, it needs a way to produce a conflict-free view of transactions for wallet operations. This is called canonicalization.

The canonical_view() method takes a ChainOracle (typically LocalChain) and produces a CanonicalView:

let view = indexed_tx_graph.canonical_view(&local_chain, chain_tip);

Conflict resolution rules:

The canonicalization algorithm resolves conflicts using this priority order:

  1. Anchors in the best chain (highest priority)
  2. If a transaction is confirmed in the current chain, it wins

  3. Deeper confirmations beat shallow ones

  4. last_seen timestamps (mempool priority)

  5. Among unconfirmed transactions, more recently seen wins

  6. This reflects current mempool state

  7. last_evicted timestamps (eviction tracking)

  8. If one conflicting tx was evicted while another remains, the surviving one wins

  9. Transitive canonicalization (ancestor priority)

  10. If a transaction's ancestors are canonical, it's canonical too
  11. This handles unconfirmed chains correctly

This sophisticated resolution strategy handles complex scenarios: - Reorgs where confirmed transactions get replaced - RBF (replace-by-fee) scenarios in mempool - Unconfirmed transaction chains - Historical conflicts

CanonicalView: The wallet interface

The resulting CanonicalView provides the clean interface wallets need:

pub struct CanonicalView<A> {
    order: Vec<Txid>,
    txs: HashMap<Txid, (Arc<Transaction>, ChainPosition<A>)>,
    spends: HashMap<OutPoint, Txid>,
    tip: BlockId,
}

This view: - Contains only canonical (non-conflicting) transactions - Orders transactions topologically (dependencies before dependents) - Provides each transaction's chain position (confirmed or unconfirmed) - Enables efficient UTXO queries - Supports balance calculations

Key methods include: - txs(): Iterate all transactions with their positions - balance(): Calculate wallet balance with customizable trust predicates - txout(): Get specific outputs with spent status - filter_outpoints(): Get relevant outputs from the indexer

Incremental canonicalization

For applications that only need to process canonical transactions without building a full view, there's canonical_iter(). This lazily produces canonical transactions one at a time, avoiding the overhead of creating the full view.

Indexers: Knowing What's Yours¶

The Indexer trait provides an abstract interface for determining which transaction outputs belong to the wallet:

pub trait Indexer {
    type ChangeSet;

    fn index_txout(&mut self, outpoint: OutPoint, txout: &TxOut) -> Self::ChangeSet;
    fn is_tx_relevant(&self, tx: &Transaction) -> bool;
    fn apply_changeset(&mut self, changeset: Self::ChangeSet);
    // ...
}

This abstraction allows different indexing strategies, but the crate provides two main implementations:

SpkTxOutIndex: Script Pubkey Indexing¶

SpkTxOutIndex is the lower-level indexer. It maintains a set of script pubkeys (spks) and tracks which outputs match them:

pub struct SpkTxOutIndex<I> {
    spks: BTreeMap<I, ScriptBuf>,
    txouts: BTreeMap<(I, OutPoint), TxOut>,
    // ...
}

The generic I parameter is an application-defined index, typically something like (KeychainKind, u32) representing a keychain and derivation index.

Key features: - Tracks which spks are "used" (have received funds) - Provides iterators over unused spks (for gap limit adherence) - Maintains all matched outputs even if spent - Returns relevant outputs efficiently by index

This indexer is ideal when you have a static set of addresses to watch or want fine-grained control over indexing.

KeychainTxOutIndex: Descriptor-Based Indexing¶

KeychainTxOutIndex is the higher-level indexer designed for descriptor-based wallets:

pub struct KeychainTxOutIndex<K> {
    spk_txout_index: SpkTxOutIndex<(K, u32)>,
    keychains: BTreeMap<K, Descriptor<DescriptorPublicKey>>,
    last_revealed: BTreeMap<K, u32>,
    lookahead: u32,
    // ...
}

It manages multiple keychains (external, internal, custom) and handles descriptor derivation automatically.

Key concepts:

  • Keychains: Different derivation paths (e.g., receiving vs change addresses)
  • Revealed spks: Addresses explicitly given to users or applications
  • Lookahead: Additional addresses derived beyond revealed to catch incoming payments
  • Last revealed index: Tracks the highest index explicitly requested per keychain

The lookahead strategy:

When you call reveal_next_spk() to get a new address, the indexer actually derives lookahead additional addresses behind the scenes. This ensures that if someone sends to an address you haven't explicitly revealed yet (address reuse, gaps in usage), you'll still detect the transaction.

Unbounded spk iterators:

For blockchain scanning, the indexer provides revealed_spks(), unused_spks(), and all_spks() iterators. These are unbounded—they continue deriving new addresses as you iterate, making it easy to scan large address ranges:

for (index, spk) in keychain_index.revealed_spks(KeychainKind::External) {
    // Scan blockchain for this spk
}

Design rationale:

Why separate SpkTxOutIndex from KeychainTxOutIndex? Because not all wallets use descriptors. By keeping the low-level indexer separate, the crate supports: - Multi-sig setups with pre-defined addresses - Lightning wallets with specific output scripts - Custom indexing strategies - Hardware wallets that provide addresses externally

IndexedTxGraph: Synchronization Guarantee¶

IndexedTxGraph combines a TxGraph with an Indexer and enforces that they stay synchronized:

pub struct IndexedTxGraph<A, I> {
    pub index: I,
    graph: TxGraph<A>,
}

The synchronization guarantee:

Every transaction or output inserted into the graph is automatically fed through the indexer. This ensures you never end up in a state where: - The graph contains transactions the indexer doesn't know about - The indexer tracks outputs that aren't in the graph

Convenient batch operations:

IndexedTxGraph provides higher-level methods that combine graph and index operations:

// Insert only transactions relevant to the indexer
pub fn batch_insert_relevant(&mut self, txs: impl IntoIterator<Item = Transaction>);

// Process an entire block efficiently
pub fn apply_block_relevant(&mut self, block: &Block, height: u32);

// Insert unconfirmed transactions with timestamp
pub fn batch_insert_unconfirmed(&mut self, txs: impl IntoIterator<Item = (Transaction, u64)>);

These methods handle the common workflow: receive blockchain data, filter for relevance, insert into graph, update index, return combined ChangeSet.

Why this matters:

The synchronization guarantee means application developers can't make mistakes. You can't forget to update the index after inserting a transaction, or vice versa. The IndexedTxGraph enforces correct usage at the type level.

How It All Fits Together¶

Let's walk through a typical wallet workflow to see how these components interact:

1. Initial Setup¶

// Create a LocalChain with the genesis checkpoint
let (mut local_chain, chain_changeset) = LocalChain::from_genesis(genesis_hash, 0);

// Create a KeychainTxOutIndex with descriptors
let mut keychain_index = KeychainTxOutIndex::new(10); // 10 address lookahead
keychain_index.insert_descriptor(KeychainKind::External, external_descriptor);
keychain_index.insert_descriptor(KeychainKind::Internal, internal_descriptor);

// Combine into an IndexedTxGraph
let mut indexed_graph = IndexedTxGraph::new(keychain_index);

2. Revealing Addresses¶

// Get the next receiving address
let ((keychain, index), spk) = indexed_graph.index
    .reveal_next_spk(KeychainKind::External);

// Convert to address and give to user
let address = Address::from_script(&spk, network);

Behind the scenes, this also derived lookahead additional addresses for scanning.

3. Blockchain Sync¶

// Get all spks to scan for (revealed + lookahead)
let spks: Vec<ScriptBuf> = indexed_graph.index
    .all_spks()
    .map(|(_, spk)| spk)
    .collect();

// Fetch blockchain data from your source (electrum, esplora, etc.)
let scan_result = blockchain_client.scan(spks);

// Apply new chain updates
let chain_changeset = local_chain.apply_update(scan_result.chain_update)?;

// Insert relevant transactions
let graph_changeset = indexed_graph.batch_insert_relevant(scan_result.transactions);

4. Getting Wallet State¶

// Create a canonical view
let chain_tip = local_chain.tip();
let view = indexed_graph.canonical_view(&local_chain, chain_tip);

// Calculate balance
let balance = view.balance(
    &local_chain,
    chain_tip.block_id(),
    |_, tx| true, // Trust all transactions
    |spk| indexed_graph.index.index_of_spk(spk).is_some(),
);

println!("Confirmed: {}", balance.confirmed);
println!("Pending: {}", balance.trusted_pending);

// List unspent outputs
for (index, txout) in view.filter_outpoints(&local_chain, chain_tip.block_id()) {
    if !txout.spent_by.is_some() && txout.is_confirmed() {
        println!("UTXO: {:?} - {} sats", index, txout.txout.value);
    }
}

// Iterate transactions
for (tx, position) in view.txs() {
    match position {
        ChainPosition::Confirmed { anchor, .. } => {
            println!("Confirmed tx {} at height {}", tx.compute_txid(), anchor.block_id.height);
        }
        ChainPosition::Unconfirmed { .. } => {
            println!("Unconfirmed tx {}", tx.compute_txid());
        }
    }
}

5. Persisting State¶

// Collect all changesets
let changesets = vec![
    chain_changeset,
    graph_changeset,
];

// Persist however you want (SQLite, files, etc.)
db.store_changesets(changesets)?;

If the application crashes and restarts, you can rebuild state by replaying all stored changesets.

Key Design Decisions and Their Rationale¶

Why Allow Conflicting Transactions?¶

The decision to allow conflicts in TxGraph rather than forcing immediate resolution is fundamental. The rationale:

  1. You might not have enough information: When you first observe conflicting mempool transactions, you don't know which will confirm (if any).

  2. Conflicts are temporary but history is permanent: Once a reorg happens, the conflicts are valuable historical data for understanding what occurred.

  3. Different contexts need different resolutions: A transaction explorer might want to show all conflicts. A wallet needs a canonical view. The graph supports both.

  4. Deferred resolution is more efficient: Resolving conflicts requires chain context. By deferring until canonical_view(), you do the work once with full information rather than repeatedly with partial information.

Why Generic Anchors?¶

Making the anchor type generic (TxGraph<A>) rather than using a concrete type enables:

  1. Minimal overhead for simple use cases: Use BlockId if you only need height/hash.

  2. Rich metadata for complex use cases: Add timestamps, merkle proofs, multiple confirmation sources, confidence scores.

  3. Application-specific requirements: Different applications need different confirmation semantics. The crate doesn't impose a one-size-fits-all solution.

  4. Future compatibility: New confirmation mechanisms (sidechains, lightning, etc.) can be supported without changing the core structures.

Why Separate SpkTxOutIndex and KeychainTxOutIndex?¶

This separation might seem like unnecessary complexity, but it provides:

  1. Flexibility for non-descriptor wallets: Not everything uses descriptors. Multi-sig coordinators, lightning wallets, and custom setups can use SpkTxOutIndex directly.

  2. Clear separation of concerns: Descriptor logic (derivation, caching) is isolated in KeychainTxOutIndex. The base indexing logic is in SpkTxOutIndex.

  3. Composability: You could build alternative high-level indexers (e.g., for miniscript, taproot trees) on top of SpkTxOutIndex.

  4. Testing and reasoning: The base indexer is simpler and easier to test thoroughly.

Why ChangeSet Pattern Instead of Direct Persistence?¶

The ChangeSet pattern imposes a bit of extra work on applications, but provides enormous benefits:

  1. Persistence agnostic: The crate doesn't need to know or care about your database, file format, or storage backend.

  2. Event sourcing: You get a complete audit trail of all state changes for free.

  3. Flexible notification: ChangeSets can drive UI updates, webhook notifications, or custom business logic.

  4. Easy merging: Multiple ChangeSets can be combined, enabling distributed/multi-device scenarios.

  5. Testability: You can verify exact ChangeSets in tests rather than just final state.

Why Monotonicity?¶

The monotone property (only append, never delete) has some costs: - Memory grows over time - You can't "forget" old transactions - Pruning requires careful design

But the benefits outweigh these costs:

  1. Correctness is easier: Impossible to end up with dangling references or inconsistent state.

  2. Recovery is trivial: Replay ChangeSets in any order and get the same result.

  3. Concurrency is simpler: Multiple threads can update safely because operations commute.

  4. Debugging is easier: You never wonder "where did that transaction go?"

For the rare cases where you need to prune old data, the crate provides mechanisms (like last_evicted timestamps) to mark data as logically deleted while maintaining structural integrity.

What the Chain Crate Supports¶

Core Functionality¶

  • ✅ Arbitrary blockchain scanning strategies (scan everything, scan subsets, incremental scanning)
  • ✅ Multiple data sources (electrum, esplora, full node, custom sources)
  • ✅ Descriptor-based key derivation with lookahead
  • ✅ Multi-keychain wallets (external, internal, custom keychains)
  • ✅ Transaction conflict resolution with configurable strategies
  • ✅ Reorganization handling with automatic anchor updates
  • ✅ Partial transaction information (outputs only, no full tx required)
  • ✅ Unconfirmed transaction chains (child-pays-for-parent)
  • ✅ Replace-by-fee (RBF) tracking and resolution
  • ✅ Flexible balance calculations with custom trust predicates
  • ✅ Transaction history with proper topological ordering
  • ✅ UTXO selection with spent status tracking
  • ✅ Mempool state tracking (first/last seen, eviction)

Advanced Use Cases¶

  • ✅ Multi-device sync (via ChangeSet merging)
  • ✅ Custom confirmation semantics (via generic anchors)
  • ✅ Historical conflict analysis (conflicts coexist in graph)
  • ✅ Efficient rescanning (via spk iterators)
  • ✅ Custom indexing strategies (via Indexer trait)
  • ✅ Event sourcing and audit trails (via ChangeSets)
  • ✅ Non-descriptor wallets (via SpkTxOutIndex directly)

What It Doesn't Support (By Design)¶

The chain crate is intentionally low-level. It doesn't include:

  • ❌ Transaction building/signing (that's in bdk_wallet)
  • ❌ Fee estimation (that's application-specific)
  • ❌ Coin selection (that's in bdk_wallet)
  • ❌ Network communication (that's in bdk_electrum, bdk_esplora, etc.)
  • ❌ Persistence implementation (that's application-specific)
  • ❌ Descriptor parsing (that's in bitcoin::descriptor)

This separation of concerns means the chain crate can focus on doing one thing exceptionally well: maintaining consistent state about blockchain data.

Conclusion¶

The bdk_chain crate represents a principled approach to a fundamental problem in Bitcoin wallet development. By embracing monotonicity, deferring conflict resolution, and separating concerns, it provides a robust foundation that higher-level libraries can build upon.

The key insights that make this design successful are:

  1. State management is hard; monotonicity makes it manageable
  2. Conflicts are reality; resolution is context-dependent
  3. Flexibility comes from abstraction (Indexer, Anchor, ChainOracle)
  4. Separation of chain/tx/index concerns enables composability
  5. ChangeSets enable persistence without imposing it

For developers building on BDK, understanding these abstractions and design decisions helps you make better architectural choices in your own applications. The chain crate gives you powerful primitives; how you compose them determines what your wallet can do.

Whether you're building a simple mobile wallet, a complex multi-sig coordinator, or a Lightning node with on-chain wallet integration, the chain crate provides the foundation you need for reliable, consistent Bitcoin state management.
